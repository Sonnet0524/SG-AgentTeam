# 框架思维指南

> 如何站在框架层面思考

---

## 思维层次

### 三个层次

```
应用层（具体项目）
   └─ knowledge-assistant实例
      - PM如何分配任务
      - Data如何写代码

框架层（通用框架）
   └─ Agent Team Framework
      - 为什么需要Agent角色
      - Agent如何交互

理论层（基础理论）
   └─ Multi-Agent System理论
      - Agent协作理论
      - 系统设计理论
```

**Research Agent工作范围**: 框架层

---

## 思维方法

### 从具体到抽象

**步骤**:

1. **观察具体案例**
   ```
   案例: knowledge-assistant项目
   - PM Agent负责协调
   - Data Agent负责数据
   - Template Agent负责模板
   ```

2. **识别共性特征**
   ```
   共性:
   - 每个Agent有明确职责
   - 职责不重叠
   - 单向依赖
   ```

3. **抽象设计模式**
   ```
   模式: Agent角色设计模式
   - 专业化原则
   - 边界隔离原则
   - 单向依赖原则
   ```

4. **形成设计思路**
   ```
   思路: Agent角色如何设计
   - 根据职责划分角色
   - 明确角色边界
   - 建立依赖关系
   ```

---

### 站在框架层面

**提问方式**:

**错误提问**（执行层）❌:
- "PM如何分配Issue #15？"
- "Data使用什么库解析YAML？"
- "测试覆盖率如何达到85%？"

**正确提问**（框架层）✅:
- "为什么需要PM角色？"
- "Agent之间如何交互？"
- "如何设计质量保证机制？"

---

### 抽象思维技巧

**技巧1: 去除具体细节**

```
执行层描述:
"PM Agent使用GitHub Issues创建任务，
分配给Data Agent，Data Agent创建分支，
编写代码，提交PR，PM Review后合并"

框架层抽象:
"Agent之间的任务分配和协作流程"
```

---

**技巧2: 提炼本质特征**

```
观察:
- PM不写代码
- Test不写代码
- 只有自己的模块才能改

本质:
- 角色职责分离
- 边界隔离原则
```

---

**技巧3: 建立通用模型**

```
具体案例:
knowledge-assistant的4个Agent

通用模型:
Agent Team的N个角色设计
```

## 思维检查

### 框架层思维检查清单

**抽象性检查**:
- [ ] 是否涉及具体技术？
- [ ] 是否涉及具体工具？
- [ ] 是否涉及具体项目？
- [ ] 是否可以应用到其他场景？

**框架性检查**:
- [ ] 是否回答"为什么"？
- [ ] 是否提出设计思路？
- [ ] 是否提炼通用原则？
- [ ] 是否具有指导价值？

**研究性检查**:
- [ ] 是否是框架设计问题？
- [ ] 是否有理论分析？
- [ ] 是否有充分论证？
- [ ] 是否有可靠结论？

## 典型案例

### 案例1: Agent角色设计

**执行层思维** ❌:
```
问题: PM Agent应该做什么？
答案: 创建Issue、分配任务、Review代码...
```

**框架层思维** ✅:
```
问题: 为什么需要协调角色？
分析: 
- 多Agent协作面临协调挑战
- 需要全局视角
- 需要决策中心

设计思路:
- 协调角色负责全局管理
- 不参与具体执行
- 提供决策支持
```

---

### 案例2: 文档交互设计

**执行层思维** ❌:
```
问题: CATCH_UP.md应该写什么？
答案: 状态、任务、约束...
```

**框架层思维** ✅:
```
问题: Agent之间如何通过文档交互？
分析:
- 文档作为交互媒介
- 支持异步通信
- 可追溯、可验证

设计思路:
- 标准化文档格式
- 分层披露信息
- 按需加载机制
```

---

### 案例3: 管理机制设计

**执行层思维** ❌:
```
问题: 如何统计Agent的工作时间？
答案: 记录启动时间和结束时间...
```

**框架层思维** ✅:
```
问题: 如何度量Agent工作量？
分析:
- 自然时间不适合Agent工作模式
- 需要与AI原生单位对齐
- Token是AI的计算单位

设计思路:
- Token-Based管理
- 工作量用Token度量
- 速度用Token/小时表示
```

## 常见误区

### 误区1: 框架层=高层管理

**错误**: 认为框架层就是管理层

**正确**: 框架层是设计思路层，管理层是执行层的一部分

---

### 误区2: 抽象=空泛

**错误**: 认为抽象就是不具体

**正确**: 抽象是提炼本质，更具指导价值

---

### 误区3: 框架层不关心实施

**错误**: 认为框架层完全脱离实际

**正确**: 框架层来自实践，指导实践
