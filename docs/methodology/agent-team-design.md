## 一、设计理念

### 1.1 Agent First vs Human First

**Agent First原则**：
- 用于Agent之间的交互文档
- 格式：结构化、简洁、机器可读
- 示例：CATCH_UP.md, ESSENTIALS.md, 索引文件

**Human First原则**：
- 用于向用户汇报的文档
- 格式：人性化、易读、有温度
- 示例：用户报告、状态总览

**设计决策**：
```
Agent间通信 → Agent First (简洁高效)
Agent→用户 → 转化为Human First (易懂友好)
```

---

### 1.2 Context窗口最小化

**目标**：最小化Agent启动时的context消耗

**策略**：
1. **分层文档**：CATCH_UP (<50行) → ESSENTIALS (<100行) → 详细指南
2. **按需披露**：只读必需信息，详细指南按需加载
3. **索引机制**：通过索引快速检索，不预加载

**效果**：
- Context使用：从~600行降至~40行
- 节省：**93%**

**核心价值**：每个字节的信息都有价值

---

### 1.3 边界隔离

**定义**：每个Agent的模块完全不重叠，零交叉修改

**目的**：
- 避免冲突
- 清晰职责
- 独立开发

**实现原则**：
- 每个Agent负责独立的目录和文件
- 通过public API交互
- 数据流单向
- 跨边界修改需审批

## 二、文档分层体系

### Level 0（必需）

| 属性 | 值 |
|------|-----|
| **大小** | <50行 |
| **加载时机** | 启动时 |
| **内容** | 当前状态、核心任务、约束条件 |
| **示例** | CATCH_UP.md |

### Level 1（按需）

| 属性 | 值 |
|------|-----|
| **大小** | <100行 |
| **加载时机** | 工作时 |
| **内容** | 职责范围、工作流程、常用命令 |
| **示例** | ESSENTIALS.md |

### Level 2（参考）

| 属性 | 值 |
|------|-----|
| **大小** | 不限 |
| **加载时机** | 按需查询 |
| **内容** | 详细指南、参考资料、经验库 |
| **示例** | guides/, knowledge-base/ |

**核心原则**：
```
Level 0 ─── 必须足够小，确保启动效率
Level 1 ─── 核心工作信息，按需加载
Level 2 ─── 详细参考，用到才查
```

## 三、Token-Based管理方法论

### 3.1 概念定义

| 单位 | 定义 | 说明 |
|------|------|------|
| **Task** | 单个工作单元 | 预估token消耗，最小粒度 |
| **Checkpoint** | 相关Tasks集合 | 功能模块子集，阶段性目标 |
| **Phase** | 功能模块 | 主要交付单元，完整功能 |

### 3.2 进度跟踪模型

```
Phase (总预估)
├── Checkpoint (子预估)
│   ├── Task (实际消耗)
│   └── Task (实际消耗)
└── Checkpoint (子预估)

Progress = 已完成token / 总预估token
Velocity = 已完成token / 工作时间
ETA = 剩余token / Velocity
```

### 3.3 优势

- ✅ 精确预估工作量（不依赖自然时间）
- ✅ 实时进度跟踪
- ✅ 准确ETC预测
- ✅ 跨项目可比性

### 3.4 注意事项

- 简单任务：准确率较高
- 复杂任务：需引入复杂度因子
- 个体差异：不同Agent效率不同

## 四、协作架构模式

### 4.1 分层协作架构

```
┌─────────────────────────────────────────────────┐
│                    Human                          │
│               (决策、监控、干预)                    │
└───────────────────┬─────────────────────────────┘
                     │ Human First
             ┌───────┴───────┐
             │  Coordinator   │ ← 协调层
             │   (PM Agent)   │
             └───────┬───────┘
                     │ Agent First
        ┌────────────┼────────────┐
        │            │            │
   ┌────┴────┐  ┌────┴────┐  ┌────┴────┐
   │ Agent A │  │ Agent B │  │ Agent C │ ← 执行层
   │(基础层) │  │(应用层) │  │(质量层) │
   └─────────┘  └─────────┘  └─────────┘
        │            │            │
        └────────────┴────────────┘
                     │
              文档化交互
            (共享上下文)
```

### 4.2 依赖方向原则

```
基础层 → 无外部依赖（被依赖）
应用层 → 依赖基础层
质量层 → 依赖所有层（测试覆盖）
管理层 → 监控所有层（不开发代码）
```

### 4.3 信息流设计

| 场景 | 信息流向 | 原则 | 格式特点 |
|------|---------|------|---------|
| Agent协作 | Agent ↔ Agent | Agent First | 结构化、简洁 |
| 状态汇报 | Agent → Human | Human First | 易读、有温度 |
| 任务分配 | PM → Agent | Agent First | 明确、可执行 |
| 问题上报 | Agent → PM | Agent First | 规范、可追溯 |

## 五、最佳实践

### 5.1 Context优化

**DO**：
- ✅ 启动时只读CATCH_UP.md
- ✅ 按需读取ESSENTIALS.md
- ✅ 使用索引检索知识库
- ✅ 详细信息放在详细文档中

**DON'T**：
- ❌ 启动时读取所有文档
- ❌ 预加载知识库内容
- ❌ 在入口文档中包含详细指南

---

### 5.2 模块边界维护

**DO**：
- ✅ 只修改自己负责的模块
- ✅ 通过public API使用其他Agent模块
- ✅ 遇到边界问题立即上报
- ✅ 保持接口稳定性

**DON'T**：
- ❌ 跨越模块边界修改
- ❌ 直接访问其他Agent的内部实现
- ❌ 破坏现有public API
- ❌ 忽略模块归属问题

---

### 5.3 Token-Based工作

**DO**：
- ✅ 准确估算任务token消耗
- ✅ 实时记录token使用
- ✅ 监控velocity趋势
- ✅ 调整预估偏差

**DON'T**：
- ❌ 使用自然日/周估算
- ❌ 忽略token消耗统计
- ❌ 不更新进度信息

---

### 5.4 知识沉淀

**DO**：
- ✅ 完成重要任务后总结经验
- ✅ 记录问题和解决方案
- ✅ 贡献到知识库
- ✅ 更新索引

**DON'T**：
- ❌ 完成任务后不总结
- ❌ 经验只留在会话中
- ❌ 知识库长期不更新

## 六、文档化交互协议

### 6.1 交互文档类型

| 类型 | 用途 | 格式 | 更新频率 |
|------|------|------|---------|
| 状态文档 | 记录当前状态 | 结构化 | 每次变更 |
| 任务文档 | 分配和跟踪任务 | 结构化 | 每个任务 |
| 报告文档 | 汇报成果 | Human First | 阶段性 |
| 决策文档 | 记录重要决策 | ADR格式 | 每个决策 |

### 6.2 交互流程

```
1. 发起方创建交互文档
2. 更新状态索引
3. 接收方读取文档
4. 接收方执行/响应
5. 更新文档状态
6. 归档或继续
```

### 6.3 Message Queue愿景

未来可实现：
- 文档即消息
- 状态驱动流转
- 自动化触发
- 无需人工协调
